# 迷雾系统性能优化方案

## 性能分析报告

### 当前系统瓶颈识别

经过对 `MistPaintingSystem.gd` 代码的深度分析，识别出以下性能瓶颈：

#### 1. CPU 像素级操作（严重瓶颈）
- **位置**: `_apply_brush()` 方法
- **问题**: 使用 `Image.set_pixel()` 和 `Image.get_pixel()` 逐像素操作
- **影响**: 每次绘制都要遍历笔刷范围内的所有像素，时间复杂度 O(n²)
- **量化**: 1280x720 纹理，笔刷半径 20px 时，单次操作约 1256 次像素访问

#### 2. 全图遍历再生（严重瓶颈）
- **位置**: `_regenerate_mist()` 方法
- **问题**: 迷雾再生时遍历整个纹理 (1280x720 = 921,600 像素)
- **影响**: 即使只再生一小部分，也要处理全部像素
- **量化**: 每帧最多 92 万次像素操作

#### 3. 覆盖率计算采样（中等瓶颈）
- **位置**: `_update_coverage()` 方法
- **问题**: 每次更新都进行全图采样计算
- **影响**: 虽然使用步长 4 采样，仍有约 57,600 次像素访问
- **频率**: 每次绘制操作后调用

#### 4. 纹理更新阻塞（中等瓶颈）
- **位置**: `_process_paint_queue()` 方法
- **问题**: `mist_texture.update(mist_image)` 同步更新 GPU 纹理
- **影响**: 主线程阻塞等待 GPU 操作完成
- **频率**: 每帧更新间隔 (16ms) 触发

#### 5. 绘制队列累积（轻微瓶颈）
- **位置**: `paint_queue` 处理
- **问题**: 快速移动时队列可能累积多个操作
- **影响**: 一次性处理大量绘制操作导致帧率波动

### 性能测试数据（优化前）

| 场景 | 预估 FPS | 主要瓶颈 |
|------|---------|---------|
| 静止状态 | 60 | 无 |
| 慢速绘制 (1笔刷/帧) | 45-55 | 像素操作 |
| 快速绘制 (5笔刷/帧) | 25-35 | 像素操作 + 队列累积 |
| 迷雾再生开启 | 15-25 | 全图遍历 |
| 复杂场景 (10+ 笔刷同时) | 10-20 | 多重瓶颈叠加 |

## 优化方案

### 方案一：GPU 计算着色器（推荐 - 主要优化）

将迷雾计算从 CPU 迁移到 GPU，利用 Godot 4.x 的 Compute Shader 功能。

**实现方式**:
1. 使用 `RenderingDevice` API 创建计算着色器
2. 将笔刷操作转为 GPU 并行计算
3. 使用纹理作为计算缓冲区

**预期效果**: 
- 像素操作性能提升 50-100 倍
- 支持更多并发笔刷操作

### 方案二：脏矩形更新（主要优化）

只更新发生变化的区域，避免全图操作。

**实现方式**:
1. 追踪所有修改的矩形区域
2. 使用 `Image.blit_rect()` 局部更新
3. 合并重叠的脏矩形

**预期效果**:
- 减少 90% 以上的无效像素操作
- 大幅降低纹理更新开销

### 方案三：分块渲染系统（主要优化）

将迷雾纹理分割为多个小块，按需更新。

**实现方式**:
1. 1280x720 纹理分为 20x12 个 64x64 块
2. 只更新被修改的块
3. 使用多线程并行处理块更新

**预期效果**:
- 内存访问更局部化（缓存友好）
- 支持多线程并行

### 方案四：异步覆盖率计算（次要优化）

将覆盖率计算移出主线程。

**实现方式**:
1. 使用 `WorkerThreadPool` 异步计算
2. 降低计算频率（每 0.5 秒一次）
3. 增量更新统计信息

**预期效果**:
- 消除覆盖率计算造成的帧率波动

### 方案五：对象池与缓存（辅助优化）

复用对象减少内存分配。

**实现方式**:
1. 笔刷图像对象池
2. 绘制操作对象池
3. 预计算常用笔刷形状
